# Technical Plan

## Status
**Phase:** Self-Review Complete

## Requirements Summary
Build a streaming podcast-note generator for a YouTube-link workflow. The app will accept a source, fetch or mock a transcript, generate a Chinese dialogue-style article, and stream the output to the frontend progressively.

The agreed architecture is:
- Frontend exposes a minimal `useDialogue()` hook that returns structured dialogue state and submit/stop/reset actions.
- Backend uses a LangGraph-style sequential workflow.
- Source fetching is abstracted, but v1 will return mock transcript data.
- Transcript normalization produces one plain text string, not timestamped segments.
- Chunking is model-assisted and creates one text chunk per intended article section.
- Each chunk is generated by one model call that emits constrained XML-like tags.
- A node-local streaming parser converts XML-like model output into structured stream events.
- Validation and chunk-level retry are required.

## Product Output Shape
The final article structure mirrors the reference document:
- A document title.
- Multiple first-level sections.
- Each section has one heading and maps to exactly one transcript chunk.
- Each section contains one to four subsections.
- Each subsection contains one to two Q&A rounds, represented as two to four turns.
- Turn text is streamed progressively while the model is generating it.

Terminology:
- `title`: whole article title.
- `section heading`: first-level article heading.
- `subsection heading`: second-level heading.
- `turn`: one speaker's utterance.

## Frontend API Design
Expose a zero-configuration hook:

```ts
export type DialogueSource =
  | { kind: "youtube"; url: string }
  | { kind: "bilibili"; url: string }
  | { kind: "article"; url?: string; title?: string; content: string };

export type DialogueStatus =
  | "idle"
  | "submitting"
  | "awaitingTitle"
  | "streamingDialogue"
  | "completed"
  | "error"
  | "aborted";

export interface DialogueSpeaker {
  id: string;
  name: string;
}

export interface DialogueTurn {
  id: string;
  speakerId: string;
  text: string;
  status: "streaming" | "completed";
}

export interface DialogueSubsection {
  id: string;
  heading: string;
  turns: DialogueTurn[];
  status: "streaming" | "completed";
}

export interface DialogueSection {
  id: string;
  chunkIndex: number;
  heading: string;
  subsections: DialogueSubsection[];
  status: "streaming" | "completed";
}

export interface DialogueState {
  status: DialogueStatus;
  title: string | null;
  speakers: DialogueSpeaker[];
  sections: DialogueSection[];
  error?: Error;
}

export interface SubmitDialogueInput {
  source: DialogueSource;
  targetLanguage?: "zh-CN";
}

export interface UseDialogueResult extends DialogueState {
  isLoading: boolean;
  submit: (input: SubmitDialogueInput) => Promise<void>;
  stop: () => void;
  reset: () => void;
}

export declare function useDialogue(): UseDialogueResult;
```

Notes:
- Do not include hook options such as `api`, `id`, `headers`, `body`, `initialState`, or callbacks in v1.
- Do not include `requestId` or `source` in returned state unless implementation later proves they are needed.
- `DialogueTurn.speakerId` is required.

## Frontend Stream Protocol
The frontend consumes a single SSE stream of JSON chunks. The stream should be compatible with the AI SDK UI message stream style where practical, but `useDialogue` owns the final aggregation logic.

Minimal event set:

```ts
type DialogueDataPart =
  | {
      type: "data-status";
      data: { status: "awaitingTitle" | "streamingDialogue" | "completed" };
    }
  | {
      type: "data-title";
      id: "title";
      data: { title: string };
    }
  | {
      type: "data-speakers";
      id: "speakers";
      data: { speakers: DialogueSpeaker[] };
    }
  | {
      type: "data-section";
      id: string;
      data: { chunkIndex: number; heading: string };
    }
  | {
      type: "data-subsection";
      id: string;
      data: { sectionId: string; heading: string };
    }
  | {
      type: "data-turn-start";
      id: string;
      data: { subsectionId: string; speakerId: string };
    }
  | {
      type: "data-turn-delta";
      id: string;
      data: { delta: string };
    }
  | {
      type: "data-turn-end";
      id: string;
    }
  | {
      type: "data-subsection-end";
      id: string;
    }
  | {
      type: "data-section-end";
      id: string;
    }
  | {
      type: "data-section-reset";
      id: string;
    };
```

Aggregation rules:
- `data-title` sets `state.title`.
- `data-speakers` sets `state.speakers`.
- `data-section` appends a section with `status: "streaming"`.
- `data-subsection` appends a subsection to its parent section with `status: "streaming"`.
- `data-turn-start` appends an empty turn with `status: "streaming"`.
- `data-turn-delta` appends `delta` to the matching turn's `text`.
- `data-turn-end` marks the turn as `completed`.
- `data-subsection-end` marks the subsection as `completed`.
- `data-section-end` marks the section as `completed`.
- `data-section-reset` removes that section and all child state before a retry replay.

ID policy:
- IDs are generated by backend code, never by the model.
- Use stable deterministic IDs per chunk attempt:
  - `section-${chunkIndex}`
  - `section-${chunkIndex}-subsection-${subsectionIndex}`
  - `section-${chunkIndex}-subsection-${subsectionIndex}-turn-${turnIndex}`
- Stable IDs allow clean reset and replay on retry.

## Backend Core Types

```ts
export interface NormalizedTranscript {
  sourceType: DialogueSource["kind"];
  sourceTitle?: string;
  language?: string;
  text: string;
}

export interface TranscriptChunk {
  chunkIndex: number;
  text: string;
}
```

Internal graph events:

```ts
type GraphEvent =
  | { type: "status.changed"; status: "awaitingTitle" | "streamingDialogue" | "completed" }
  | { type: "title.generated"; title: string }
  | { type: "speakers.generated"; speakers: DialogueSpeaker[] }
  | { type: "section.started"; sectionId: string; chunkIndex: number; heading: string }
  | { type: "subsection.started"; subsectionId: string; sectionId: string; heading: string }
  | { type: "turn.started"; turnId: string; subsectionId: string; speakerId: string }
  | { type: "turn.delta"; turnId: string; delta: string }
  | { type: "turn.completed"; turnId: string }
  | { type: "subsection.completed"; subsectionId: string }
  | { type: "section.completed"; sectionId: string }
  | { type: "section.reset"; sectionId: string }
  | { type: "run.completed" };
```

The stream adapter maps `GraphEvent` to `DialogueDataPart`.

## Workflow Steps
Use a sequential workflow for v1:

1. `fetchSource`
   - Input: `{ source: DialogueSource }`
   - Output: `{ transcript: NormalizedTranscript }`
   - v1 returns mock transcript data.
   - Do not implement YouTube fetching yet.

2. `generateTitle`
   - Input: `{ transcript: NormalizedTranscript }`
   - Output: `{ title: string }`
   - Emit `title.generated` as soon as complete.

3. `identifySpeakers`
   - Input: `{ transcript: NormalizedTranscript }`
   - Output: `{ speakers: DialogueSpeaker[] }`
   - Emit `speakers.generated` before any turn events.
   - Speaker objects only contain `id` and `name`.
   - Provide fallback speakers if model output is ambiguous.

4. `chunkTranscript`
   - Input: `{ transcript: NormalizedTranscript }`
   - Output: `{ chunks: TranscriptChunk[] }`
   - Uses model-assisted semantic chunking.
   - Output chunks are plain text only; no headings or segment metadata.
   - Each chunk should correspond to one future first-level section.

5. For each chunk in order, call `streamSectionDialogue` with retry.
   - Input: `{ chunk: TranscriptChunk; speakers: DialogueSpeaker[]; retryContext?: RetryContext }`
   - Output: graph custom events while streaming.
   - Uses one model call per chunk.
   - Model output is constrained XML-like text.
   - Node-local parser emits turn deltas as text arrives.

6. Emit completion.

## Model-Assisted Chunking
Prompt intent:

```txt
Split the transcript into text chunks suitable for article sections.

Rules:
- Each chunk should map to exactly one first-level article section.
- Each chunk should cover one coherent topic.
- Each chunk should later support one section heading.
- Each chunk should later support one to four subsection headings.
- Each subsection should later support one to two Q&A rounds.
- Preserve source order.
- Do not create headings, summaries, or analysis.
- Return only the chunks.
```

Implementation note:
- Prefer structured output for the chunking node so it returns validated `{ chunks: string[] }`.
- Convert to `TranscriptChunk[]` by assigning `chunkIndex` in backend code.

## Stream Section Dialogue Node
This node uses one model call per chunk and requires XML-like output.

Allowed model output:

```xml
<section heading="...">
  <subsection heading="...">
    <turn speakerId="...">...</turn>
  </subsection>
</section>
```

Rules:
- Exactly one `<section>`.
- One to four `<subsection>` children.
- Each subsection contains two to four `<turn>` children.
- `speakerId` must be one of the identified speakers.
- No markdown fences.
- No text outside allowed tags.
- No additional tags.
- No additional attributes.
- Text inside `turn` must escape literal `<` and `>` if needed.

Parser states:

```ts
type ParserState =
  | "awaiting-section"
  | "inside-section"
  | "inside-subsection"
  | "inside-turn"
  | "completed";
```

Valid transitions:

```txt
awaiting-section
  -> <section heading=""> -> inside-section

inside-section
  -> <subsection heading=""> -> inside-subsection
  -> </section>            -> completed

inside-subsection
  -> <turn speakerId="">    -> inside-turn
  -> </subsection>          -> inside-section

inside-turn
  -> text                   -> emit turn.delta
  -> </turn>                -> inside-subsection
```

Any invalid transition throws a retryable validation error.

## Validation And Retry
Validation happens in two layers.

Streaming validation:
- First non-whitespace output must be `<section>`.
- Allowed tags only: `section`, `subsection`, `turn`.
- Required attributes must exist and be non-empty.
- Unknown attributes fail validation.
- `speakerId` must be in the speaker table.
- Section count cannot exceed one.
- Subsection count cannot exceed four.
- Text outside `turn` tags is invalid, except whitespace.

Final validation:
- Final parser state must be `completed`.
- Section must contain at least one subsection.
- Each subsection must contain at least two turns.
- Each subsection should contain no more than four turns.
- Each turn text must be non-empty after trimming.

Validation error codes:

```ts
type SectionDialogueValidationErrorCode =
  | "unexpected_text"
  | "unexpected_tag"
  | "unknown_tag"
  | "missing_attribute"
  | "invalid_attribute"
  | "empty_heading"
  | "unknown_speaker"
  | "too_many_subsections"
  | "empty_turn"
  | "unclosed_tag"
  | "incomplete_section";
```

Retry behavior:
- Retry at chunk level.
- `maxRetries` should be 2 or 3.
- First attempt streams normally.
- On validation failure, abort the model stream, emit `section.reset`, and regenerate the whole section for the same chunk.
- Retry prompt includes the validation failure reason and allowed speaker IDs.
- If retries are exhausted, emit an error response and set frontend state to `error`.
- Do not skip failed chunks in v1.

## Model Integration And Testing
Production workflow code must not branch into a mock model when `OPENAI_API_KEY` is absent. Model dependencies are injected at workflow construction time.

Runtime model:
- Use LangChain.js OpenAI integration for OpenAI.
- Construct the production model from Worker env, e.g. `ChatOpenAI({ model: "gpt-5-mini", apiKey: env.OPENAI_API_KEY })` or equivalent official LangChain initializer.
- Missing `OPENAI_API_KEY` is a configuration error for the real streaming endpoint.

Testing model:
- Tests inject fake chat models/runnables from the test layer.
- Prefer `FakeListChatModel` from `@langchain/core/utils/testing` for deterministic invoke and streaming behavior.
- Parser tests can bypass the model and feed token chunks directly into the node-local parser.
- Workflow tests should assert graph/custom events using injected fake model output, not production environment branches.

## Cloudflare Worker And LangGraph Notes
LangGraph JS supports streaming graph outputs via stream modes such as `custom`, `updates`, and `streamEvents`. Use `custom` or custom callback events for business events from graph nodes. Do not expose raw LangGraph `updates` or `values` to the frontend.

The Worker response should be a streaming `Response` using Web Streams/SSE. The LangGraph package/runtime compatibility with Cloudflare Workers still needs a spike during implementation. If incompatibility appears, keep the graph abstraction but implement the v1 workflow as a small sequential async generator in the Worker.
